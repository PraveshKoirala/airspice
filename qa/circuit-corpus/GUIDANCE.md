# Circuit Refinement Corpus — Prompts & Guidance

This directory is the durable record of the automated circuit-refinement loop:
the prompts fed to the AirSpice AI Assistant, the guidance behind them, the
harness that runs them against the real webapp, and the results.

## Goal

Stress-test the end-to-end product (describe → build → schematic → simulate →
firmware) by having the in-app agent build up to 100 complex circuits, ≥80 with
distinct MCUs and firmware. Every failure that is a *webapp bug* gets fixed;
circuit-level non-convergence is recorded but not treated as a bug.

## What the engine actually supports (ground truth)

From `packages/air-ts/src/registry/data.generated.ts`:

- **SPICE-simulatable primitives:** resistor, capacitor, diode, LED (as a
  `generic_load`/diode), MOSFET, LDO, voltage/current sources.
- **MCUs (4):** `ESP32-C3` (ADC vref 3.3 V, 12-bit), `ESP32-WROOM-32` (3.3 V),
  `ATmega328P` (5.0 V, 10-bit), `STM32F103` (3.3 V). MCUs are
  `spice_supported: false` — their pins are not sources in the netlist.
- **Firmware task steps:** `read_adc`, `convert`, `write_gpio`, `delay`, `log`.
- **ADC bindings** must reference a pin whose `function` matches the channel
  (e.g. `ADC1_CH4`), and the bound net must stay at or below the ADC reference
  or validation emits `ADC_INPUT_EXCEEDS_VREF`.

## Prompt design principles

The prompts (in `prompts.json`, generated by `../gen-corpus.mjs`) are written as
a real engineer's natural-language requests — the actual product surface — not
hand-authored XML. Guidance baked into every prompt:

1. **Stay inside the simulatable envelope.** Analog front-ends use only R/C/
   diode/MOSFET/LDO/sources so ngspice can actually solve them. We deliberately
   avoid discrete ICs the registry does not model (555 timers, logic counters,
   op-amps) — those stage and render fine but do not converge in SPICE, which is
   a modelling gap, not a webapp bug.
2. **Keep ADC inputs under vref.** Every sensor prompt scales the sensed voltage
   below the named MCU's ADC reference (via a divider) so firmware ADC bindings
   validate.
3. **Spread MCUs evenly.** 84 MCU circuits, 21 per supported part, across 10
   firmware templates (battery monitor, thermistor, photocell, potentiometer,
   dual-ADC, rail self-monitor, LED blinker, MOSFET load switch, I2C node,
   heartbeat+sense).
4. **Distinguish sensing from actuation.** Sensor circuits (ADC front-ends) have
   a fully-determined probed net → `expectSimPass: true`. Pure actuators
   (blinkers, MOSFET switches) drive nets from the *non-simulated* MCU, so a sim
   non-pass is expected and not asserted (`expectSimPass: false`).

## Per-spec expectations (asserted by the harness)

| field | meaning |
|---|---|
| `expectStaged` | agent must stage a gated, valid design (all specs) |
| `expectFirmware` | Firmware tab must render a project (MCU specs) |
| `expectSimPass` | sim should pass (sensor front-ends only) |

## Harness methodology (`../run-circuits.mjs`)

Each spec runs in an isolated browser context: load `/project`, force a blank
design via the `window.__airSetXml` dev hook, type the prompt, wait for the
agent run to finish, click **Apply** *only if the button is actually visible*
(this is how the earlier Apply-button CSS-crush bug is caught — a zero-height
button is invisible to Playwright), run the simulation, and for MCU specs open
the Firmware tab and confirm a project renders. It records page errors, console
errors (ngspice engine diagnostics filtered out), the produced XML, the
transcript tail, and a **verdict**:

- `ok` — staged, visible Apply, applied, rendered, firmware present (if
  expected), sim passed (if expected).
- `bug:*` — a webapp defect: `page-error`, `no-visible-apply`, `apply-failed`,
  `blank-schematic`, `firmware-missing`. **These get fixed.**
- `review:sim-fail` / `review:console-error` — needs a human/triage look.
- `agent-fail:*` — the agent never staged (prompt/model/budget), not necessarily
  an app bug.
